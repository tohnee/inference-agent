# inference-agent 架构审查（推理优化支撑能力）

日期：2026-04-17

## 审查范围

- 根目录架构与模块职责划分
- `auto-profiling` 运行时编排与状态管理
- 评估器（`scorer.py`）在“正确性优先”下的判定逻辑
- 现有测试对关键路径的覆盖情况

## 总体结论

整体架构**合理**，并且具备支撑“可持续推理优化”的核心能力。尤其是在以下方面比较成熟：

1. **知识层与执行层解耦**：三个 `*-opt-skill` 模块负责方法论与路线，`auto-profiling` 负责可执行闭环。
2. **正确性闸门清晰**：支持 exact-parity 与 bounded-tolerance 两种模式，明确“先正确再提速”。
3. **可恢复与可审计**：通过 `.auto-profiling/` 产物实现会话恢复、实验留痕、handoff。
4. **多场景可扩展**：内建 `e2e-inference` / `llm-serving` / `cuda-kernel` 三条 lane 的路由元数据。

从“是否能支持推理优化”角度看，答案是：**能支持，而且设计上偏工程化、偏长期迭代友好**。

## 架构优点（代码级证据）

### 1) 分层清楚：知识路由 vs 运行时引擎

- 根 README 将系统划分为三个优化知识模块 + 一个运行时 harness，角色边界清晰，避免把策略与执行耦在一起。
- `auto-profiling/README.md` 进一步把 Planner / Generator / Evaluator / Handoff 四个角色拆开，符合可演进的 agentic 优化流程。

### 2) 运行时具备“闭环”而不是“单次脚本”

`runner.py` 把真实优化流程抽象成：

- 读取 `aim` 合约
- 初始化工作区
- 执行 baseline / candidate / evaluate / loop
- 记录状态与实验日志
- 输出 evaluator 报告与下一步 handoff

这意味着它不是一次性 benchmark 脚本，而是可持续优化的流程引擎。

### 3) 正确性优先机制完善

`scorer.py` 中：

- exact-parity 默认严格一致
- bounded-tolerance 需要显式阈值并检查逻辑/算法等价条件
- baseline 先过 exactness 才允许比较 candidate

这可以有效避免“吞精度换性能”的隐性回归，是推理优化系统必须具备的约束。

### 4) 观测与环境自描述能力较强

`runner.py` 在环境探测中收集 shell、包管理器、工具链路径、依赖版本、可选 `vllm collect-env` 摘要，有助于复现实验和跨机迁移。

### 5) 三场景 lane 元数据化

`SCENARIO_LANES` 将场景与目标模块、推荐 skill route、推荐 aim 模板绑定，能帮助团队在不同优化问题间快速路由，降低“盲目优化”的概率。

## 主要风险与改进建议

### 风险 A：`aim` 解析器是“轻量 Markdown 约定”，鲁棒性有限

当前 `parse_aim_markdown` 基于简单行规则解析，适合模板化输入，但对复杂 Markdown、缩进差异、非法值提示不够友好。

**建议**：

- 增加 `aim` schema 校验（必填项、枚举值、路径存在性）
- 报错时输出“字段名 + 示例修复”

### 风险 B：候选执行流程与 baseline 指令复用较强

`run_contract` 对 baseline 与 candidate 共用同一套命令字段；虽然简化了配置，但某些真实项目中 candidate 可能需要独立流程（不同 profile 参数、不同数据子集）。

**建议**：

- 引入 `candidate_*` 命令可选覆盖字段
- 若未配置则回退 baseline 字段，保持兼容

### 风险 C：缺少稳定性统计与多次采样判定

当前 `compare_runs` 基于单次指标比较，缺少置信度或抖动抑制机制，容易受瞬时噪声影响。

**建议**：

- 在运行时支持 `n` 次重复 + 聚合（median/p95）
- 增加最小可感知改进阈值（例如 `min_relative_improvement`）

### 风险 D：并发实验与资源隔离能力尚未体现

现有流程对单序列实验友好，但在多模型/多卡并行调优、共享机器配额下，尚未看到队列或资源仲裁设计。

**建议**：

- 增加 experiment queue 与资源标签（GPU ID、NUMA、host）
- 记录资源占用快照，避免“测量污染”

## 是否能支持推理优化（结论细化）

### 能支持的层面

- **流程治理**：baseline/candidate/evaluator/handoff 的闭环完整
- **正确性治理**：exactness gate 明确且默认严格
- **场景覆盖**：e2e、serving、kernel 三层问题均有入口
- **工程可持续性**：状态与日志可恢复，可交接

### 仍需加强的层面（若目标是“生产级大规模优化平台”）

- 更强的合约校验与错误提示
- 统计稳健性（抗噪比较）
- 并发调度与资源管理
- 与 CI/CD / 远程执行器的更深集成

## 最终判断

对当前仓库定位（“推理优化知识库 + 最小可运行 harness”）来说，架构设计是**合理且方向正确**的；
对“是否能支持推理优化”这一问题，结论是**可以支持，并且已经具备正确性优先的工程基础**。

若按上述风险点做增强，这个系统可以从“可用的优化框架”进一步进化为“可规模化的优化平台”。
